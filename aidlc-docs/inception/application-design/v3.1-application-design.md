# Diseño de Aplicación — Abstracción Arquitectónica v3.1

## Resumen Ejecutivo

Refactoring arquitectónico del Portal EA v3.0 para introducir una arquitectura hexagonal (Ports & Adapters) que desacopla las dependencias externas (Supabase, Gemini, Groq) del código de negocio. El objetivo es habilitar una migración futura a AWS sin romper funcionalidad.

**Patrón Arquitectónico**: Hexagonal Architecture (Ports & Adapters)  
**Estrategia**: Branch by Abstraction — Supabase sigue activo, solo cambia la estructura interna  
**Impacto en UX**: Cero — refactoring 100% interno

---

└───────────────────────────┬─────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    CORE LAYER (nuevo)                             │

│  │   PORTS     │  │  CONTAINER   │  │    CROSS-CUTTING       │ │
│  │ (interfaces)│  │  (DI/SL)     │  │ Logger│Metrics│Errors   │ │
│  └─────────────┘  └──────────────┘  └────────────────────────┘ │
└───────────────────────────┬─────────────────────────────────────┘
                            │ implements

┌─────────────────────────────────────────────────────────────────┐
│                    ADAPTERS LAYER (nuevo)                         │

## Decisiones Arquitectónicas Clave

### ADR-001: Hexagonal Architecture (Ports & Adapters)

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | 15+ archivos con dependencia directa a Supabase SDK |
| **Decisión** | Adoptar Hexagonal Architecture con Ports (interfaces) y Adapters (implementaciones) |
| **Alternativas** | Clean Architecture (más capas), Onion Architecture (similar), Service Layer simple |
| **Consecuencias** | (+) Desacoplamiento total, (+) testeable, (+) migratable, (-) más archivos, (-) indirección |

### ADR-002: Result Type vs Exceptions

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Manejo de errores inconsistente (try/catch mezclado con returns de null) |
| **Decisión** | Usar `Result<T, E>` discriminated union para todos los retornos de ports |
| **Alternativas** | Throw exceptions (actual), Either monad (fp-ts), tuple [error, data] |
| **Consecuencias** | (+) Type-safe, (+) explicit error handling, (+) exhaustive switch, (-) más verboso |

### ADR-003: Container Propio vs Framework DI

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Se necesita resolución de dependencias sin agregar peso al bundle |
| **Decisión** | Implementar Service Locator ligero (~50 líneas) sin dependencias externas |
| **Alternativas** | tsyringe, inversify, typedi, awilix |
| **Consecuencias** | (+) Zero bundle impact, (+) sin decorators, (+) simple, (-) sin auto-discovery |

### ADR-004: Pino como Logger

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Se necesita logging estructurado con $0 de costo |
| **Decisión** | Pino — fastest JSON logger para Node.js, pino-pretty para desarrollo |
| **Alternativas** | Winston (más pesado), Bunyan (sin mantenimiento), console.log (no estructurado) |
| **Consecuencias** | (+) ~2KB bundle, (+) JSON nativo, (+) child loggers, (-) nueva dependencia |

### ADR-005: Métricas en Memoria

| Aspecto | Detalle |
|---------|---------|
| **Contexto** | Se necesitan métricas de rendimiento con $0 de costo |
| **Decisión** | Recolección en memoria expuesta vía `/api/metrics` en formato Prometheus |
| **Alternativas** | OpenTelemetry Collector (requiere infra), StatsD (requiere servidor), Datadog ($$$) |
| **Consecuencias** | (+) $0, (+) scrape-able por Prometheus/Grafana Cloud free, (-) se pierde en restart, (-) solo una instancia |
