# Componentes — Abstracción Arquitectónica v3.1

## Resumen

La arquitectura hexagonal (Ports & Adapters) introduce las siguientes capas de componentes nuevos sobre la aplicación existente.

---

## 1. Ports (Interfaces / Contratos)

### 1.1 Repository Ports

| Componente | Propósito | Dominio |
|------------|-----------|---------|
| `SessionRepository` | CRUD + queries de sesiones de evaluación | sessions |
| `RespondentRepository` | Registro, búsqueda, eliminación de encuestados | respondents |
| `ResponseRepository` | Almacenamiento y consulta de respuestas (upsert batch) | responses |
| `DimensionRepository` | Consulta de dimensiones con preguntas | dimensions |
| `QuestionRepository` | Consulta de preguntas por dimensión | questions |
| `InstrumentRepository` | CRUD instrumentos y versiones | instruments, instrument_versions |
| `ProfileRepository` | Gestión de perfiles de usuario y roles | profiles |
| `TenantRepository` | Gestión de tenants y límites | tenants |
| `ViewerLinkRepository` | Creación y validación de enlaces compartidos | viewer_links |
| `UsageLogRepository` | Registro de eventos de uso (análisis IA, etc.) | usage_logs |
| `AnalysisRepository` | Almacenamiento de análisis IA generados | session_analyses |

### 1.2 Auth Ports

| Componente | Propósito |
|------------|-----------|
| `AuthProvider` | Operaciones de autenticación (sign in, sign out, get user, refresh) |
| `AuthGuard` | Verificación de roles y permisos (isAuthenticated, hasRole, getCurrentProfile) |
| `AuthMiddleware` | Hook para middleware de Next.js (protección de rutas) |

### 1.3 AI Ports

| Componente | Propósito |
|------------|-----------|
| `AIProvider` | Generación de completions (generateCompletion, isAvailable, getModelInfo) |
| `AIProviderChain` | Orquestación de múltiples providers con failover configurable |

### 1.4 Observability Ports

| Componente | Propósito |
|------------|-----------|
| `Logger` | Logging estructurado (info, warn, error, debug) con contexto |
| `MetricsCollector` | Recolección de métricas (latencia, errores, throughput) |

## 2. Adapters (Implementaciones Concretas)

### 2.1 Supabase Adapters (Implementación Actual)

| Componente | Implementa | Notas |
|------------|-----------|-------|
| `SupabaseSessionRepository` | SessionRepository | Usa supabase.from('sessions') |
| `SupabaseRespondentRepository` | RespondentRepository | Incluye lógica de upsert + completed |
| `SupabaseResponseRepository` | ResponseRepository | Batch upsert con question_id |
| `SupabaseDimensionRepository` | DimensionRepository | Join con questions |
| `SupabaseQuestionRepository` | QuestionRepository | Query por dimension_id |
| `SupabaseInstrumentRepository` | InstrumentRepository | Join con instrument_versions |
| `SupabaseProfileRepository` | ProfileRepository | Incluye role checking |
| `SupabaseTenantRepository` | TenantRepository | Verifica is_active + limits |
| `SupabaseViewerLinkRepository` | ViewerLinkRepository | Token generation + validation |
| `SupabaseUsageLogRepository` | UsageLogRepository | Insert con count por periodo |
| `SupabaseAnalysisRepository` | AnalysisRepository | CRUD session_analyses |
| `SupabaseAuthProvider` | AuthProvider | Envuelve @supabase/ssr auth |
| `SupabaseAuthGuard` | AuthGuard | Verifica profiles + ADMIN_EMAILS |
| `SupabaseAuthMiddleware` | AuthMiddleware | Envuelve proxy.ts actual |

## 3. Infrastructure / Cross-Cutting

| Componente | Propósito | Tipo |
|------------|-----------|------|
| `Container` | Service Locator / DI Registry — resuelve implementaciones | Singleton |
| `InstrumentedRepository<T>` | Decorator que agrega logging + metrics a cualquier repository | Decorator |
| `PinoLogger` | Implementación del Logger con Pino (JSON en prod, pretty en dev) | Singleton |
| `InMemoryMetricsCollector` | Recolecta métricas en memoria, expone vía /api/metrics | Singleton |
| `DomainError` | Base class para errores tipados (NotFound, Unauthorized, Conflict, etc.) | Value Object |
| `RequestContext` | Porta request-id para correlación a través de la cadena | Per-request |

---

## 4. Componentes Existentes (Sin Cambios Estructurales)

Los siguientes componentes mantienen su estructura pero **cambian sus imports** para usar el Container en vez de Supabase directamente:

| Componente | Ubicación | Cambio |
|------------|-----------|--------|
| SurveyForm | components/SurveyForm.tsx | `createClient()` → `container.resolve(RespondentRepository)` |
| AdminDashboard | app/admin/page.tsx | `supabase.from('sessions')` → `container.resolve(SessionRepository)` |
| EncuestaPage | app/encuesta/[sessionId]/page.tsx | `createServerSupabaseClient()` → server container |
| ResultsPage | app/resultados/[respondentId]/page.tsx | Idem |
| API Analysis | app/api/analysis/route.ts | SDKs directos → `container.resolve(AIProviderChain)` |
| Middleware (proxy.ts) | src/proxy.ts | Supabase auth → `container.resolve(AuthMiddleware)` |
| Todas las API routes | app/api/* | `createServerSupabaseClient()` → server container |
