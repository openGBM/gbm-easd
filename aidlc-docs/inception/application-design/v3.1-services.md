# Servicios — Abstracción Arquitectónica v3.1

## Filosofía de Servicios

La arquitectura tiene **dos capas de servicios** con propósitos diferentes:

1. **Application Services (Use Cases)** — `core/services/`: Orquestan operaciones multi-paso que involucran múltiples repos. Son la lógica de negocio de la aplicación. Ver detalle completo en [v3.1-application-services.md](./v3.1-application-services.md).

2. **Infrastructure Services (Cross-Cutting)** — `core/observability/`, Container, AIProviderChain: Servicios técnicos que no contienen lógica de negocio sino infraestructura transversal.

**Flujo de una operación:**
```
Consumer → Service (use case) → Repository (port) → Adapter (Supabase/AWS)
```

**Decisión de diseño (ADR-006)**: Los services se instancian directamente en los consumers (no vía Container) recibiendo repos del Container como dependencias. Esto evita agregar 5+ tokens adicionales y mantiene la simplicidad del DI.

## Servicios Cross-Cutting (Infraestructura)

### 1. Container Service

| Aspecto | Detalle |
|---------|---------|
| **Propósito** | Registry central que resuelve implementaciones según configuración de entorno |
| **Tipo** | Singleton (uno por entorno: server / client) |
| **Configuración** | Variables de entorno: `PROVIDER`, `AI_PROVIDERS` |
| **Responsabilidades** | Registrar factories, resolver instancias, manejar scopes |

**Comportamiento:**
- En **server-side**: Resuelve repositorios con cliente Supabase server + admin
- En **client-side**: Resuelve repositorios con cliente Supabase browser
- La resolución es lazy (se crea la instancia al primer `resolve()`)
- Los singletons se cachean por la vida del proceso

**Dos containers separados:**
```
ServerContainer → para app/api routes, server components, middleware
ClientContainer → para client components ('use client')
```

### 2. Observability Service (Decorator Pattern)

| Aspecto | Detalle |
|---------|---------|
| **Propósito** | Instrumentar automáticamente cada operación de repositorio/AI con logging y métricas |
| **Tipo** | Decorator (wrappea cualquier port) |
| **Componentes** | InstrumentedRepository, PinoLogger, InMemoryMetricsCollector |
| **Responsabilidades** | Medir latencia, contar errores, correlacionar requests |

**Flujo de una operación instrumentada:**
```
Consumer llama repository.findById("abc")
  → InstrumentedRepository intercepta
    → logger.debug("SessionRepository.findById start", { id: "abc" })
    → const start = performance.now()
    → const result = await inner.findById("abc")
    → const duration = performance.now() - start
    → metrics.recordLatency("SessionRepository.findById", duration)
    → if (!result.ok) metrics.recordError("SessionRepository.findById", result.error.code)
    → logger.info("SessionRepository.findById complete", { duration, ok: result.ok })
    → return result
```

### 3. AI Provider Chain Service

| Aspecto | Detalle |
|---------|---------|
| **Propósito** | Orquestar múltiples AI providers con failover automático |
| **Tipo** | Singleton |
| **Configuración** | `AI_PROVIDERS=gemini,groq` (orden = prioridad de failover) |
| **Responsabilidades** | Intentar provider primario, fallback a secundario, registrar métricas por provider |

**Flujo de failover:**
```
AIProviderChain.generate(prompt)
  → GeminiProvider.isAvailable()? 
    → Sí: GeminiProvider.generateCompletion(prompt)
      → Éxito: return result
      → Error: log warn, continue to next
    → No: skip, continue to next
  → GroqProvider.isAvailable()?
    → Sí: GroqProvider.generateCompletion(prompt)
      → Éxito: return result
      → Error: return ServiceUnavailableError
    → No: return ServiceUnavailableError
```

### 4. Request Context Service

| Aspecto | Detalle |
|---------|---------|
| **Propósito** | Propagar request-id y metadata a través de toda la cadena de operaciones |
| **Tipo** | Per-request (AsyncLocalStorage en Node.js) |
| **Datos** | requestId, userId, tenantId, startTime |
| **Responsabilidades** | Crear contexto en middleware, disponible en toda la cadena |

## Patrones de Comunicación

### Consumer → Container → Port → Adapter

```
┌──────────────┐       ┌───────────────┐       ┌──────────────┐       ┌────────────────┐
│   Consumer   │──────▶│   Container   │──────▶│     Port     │──────▶│    Adapter     │
│ (page/route) │resolve│  (registry)   │returns│ (interface)  │  impl │ (supabase/aws) │
└──────────────┘       └───────────────┘       └──────────────┘       └────────────────┘
```

### Con Observabilidad (Decorator)

```
┌──────────────┐       ┌───────────────┐       ┌─────────────────────┐       ┌────────────────┐
│   Consumer   │──────▶│   Container   │──────▶│ InstrumentedRepo    │──────▶│    Adapter     │
│ (page/route) │resolve│  (registry)   │returns│ (decorator + logs)  │ inner │ (supabase/aws) │
└──────────────┘       └───────────────┘       └─────────────────────┘       └────────────────┘
                                                        │
                                                        ▼
                                               ┌──────────────────┐
                                               │ Logger + Metrics  │
                                               └──────────────────┘
```

### Server-Side vs Client-Side

| Contexto | Container | Auth | Repos use |
|----------|-----------|------|-----------|
| Server Components | ServerContainer | Cookies-based (SSR) | service_role o anon según operación |
| API Routes | ServerContainer | Cookies-based (SSR) | service_role para admin ops |
| Client Components | ClientContainer | Browser session | anon key (RLS activo) |
| Middleware | ServerContainer | Cookies-based (SSR) | anon key |

## Endpoints Nuevos

### GET /api/metrics

| Aspecto | Detalle |
|---------|---------|
| **Propósito** | Exponer métricas recolectadas en formato Prometheus-compatible |
| **Protección** | Solo admin autenticado |
| **Formato** | text/plain (Prometheus exposition format) |
| **Contenido** | Latencias (histograma), errores (counter), throughput (counter) por operación |

### GET /api/health

| Aspecto | Detalle |
|---------|---------|
| **Propósito** | Health check profundo — verifica conectividad a DB + Auth |
| **Protección** | Público (shallow) / Admin (deep) |
| **Respuesta** | `{ status: "healthy" | "degraded" | "unhealthy", checks: {...} }` |
